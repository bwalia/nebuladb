Within-region HA

Raft-backed clustering, quorum-durable by default

Opt into cluster mode with a single env var. Built on openraft, every acknowledged write is committed by a majority before it returns.

Quorum-durable writes

A write is acknowledged only after a majority of nodes fsync it to the Raft log. It survives loss of any minority — including the leader. No data loss.

Automatic leader failover

If the leader dies, survivors elect a new one in milliseconds. Writes to a non-leader return the new leader's address so smart clients re-target instantly.

One binary, three protocols

REST, gRPC, and Postgres wire all serve the same corpus. The same node that answers a semantic search also participates in consensus.

Self-healing membership

Take a node down, bring a new one up

The cluster reconfigures itself at runtime — no write outage, no data loss — with a learner catch-up → promote → evict flow. Quorum of the current voter set stays available at every step.

1

Catch up

Add the replacement as a non-voting learner. openraft streams it a snapshot + log tail and blocks until it's current. Quorum is unchanged, so writes keep flowing.

2

Promote

Once caught up, promote the learner to a voter via joint consensus. The cluster now has its full voting strength back.

3

Evict

Remove the dead node from the quorum. Back to a healthy N-node cluster — one member brand new, zero interruption. 

# Replace dead node 2 with a fresh node 4 — one command, quorum-safe.
scripts/replace_node.sh --leader http://leader:8080 \
  --new-id 4 --new-addr node4:50052 --old-id 2

# …or drive the membership API directly:
curl -X POST   .../admin/raft/learner -d '{"node_id":4,"addr":"node4:50052"}'  # catch up
curl -X POST   .../admin/raft/voter/4                                          # promote
curl -X DELETE .../admin/raft/node/2                                           # evict
Quorum-safe by construction. A quorum of the current voter set is available throughout, so committed writes are never lost and the leader never stalls. The replacement script refuses to run if evicting the old node would break quorum.
Across regions

Active-active multi-region, automatic failover

Each bucket has a home region; regions tail each other's WAL. A failover monitor detects a dead home region and promotes the local one automatically — no operator in the loop.

Epoch-fenced ownership

A monotonic home_epoch fences out a deposed region's stale writes. When it recovers, it self-demotes to a consumer.

Autonomous detection

Continuous-unreachability grace + hysteresis. When a home region is down past the grace window, a candidate region bumps the epoch and takes over.

No oscillation

A single opt-in candidate per home, monotonic epochs (no auto-failback), and a post-promotion cooldown keep a flaky WAN from flapping ownership. 

# Mark us-west-2 a failover candidate for buckets homed in us-east-1.
NEBULA_REGION=us-west-2 \
NEBULA_REGION_FAILOVER_CANDIDATE=true \
NEBULA_REGION_FAILOVER_BUCKETS=catalog=us-east-1,orders=us-east-1 \
  nebula-server
